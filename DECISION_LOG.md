# Decision Log

One line per meaningful AI call. The point is to show the reasoning behind each step, not just list what happened.

---

| # | What I asked the AI | What it gave back | Accepted / rejected / modified — and why |
|---|---------------------|-------------------|------------------------------------------|
| 1 | Read through these two monolithic files — main.py and App.jsx. List all the bugs you can find, especially anything around concurrency, shared state, or memory leaks. Don't fix anything yet, I just want to see what we're dealing with. | Came back with 17 issues across both files. On the backend: mutable default arguments leaking shared state across calls, a blocking time.sleep inside an async function that would freeze the entire event loop, no UUID generation so concurrent requests would collide, and no input validation. On the frontend: a setInterval that was never cleared, no try/catch on any network calls, and LLM output being injected directly as raw HTML which is a straightforward XSS vulnerability. | Accepted. I went through each one manually before trusting it. The critical ones — shared state, event loop blocking, and the HTML injection — were all real and easily reproducible. I decided to fix the bugs in the monolith first before touching the structure, so each fix could be tested cleanly. |
| 2 | Fix the 8 backend bugs in main.py. Swap the blocking sleep for a thread pool executor, generate proper UUIDs for run IDs, add Pydantic v2 validators for inputs. Keep everything in one file for now. | Returned a revised main.py with all 8 fixes applied in place. | Accepted. Keeping it as a single file at this stage was intentional — I wanted to verify the core logic was solid before splitting it up. Ran it locally, tested concurrent requests, and confirmed the event loop was no longer blocking. |
| 3 | Now fix the 9 frontend bugs in App.jsx. Add try/catch to the fetch calls, clear the polling interval on component unmount, and replace dangerouslySetInnerHTML with safe text rendering. | Returned a revised App.jsx with proper error handling, useEffect cleanup, and secure output rendering. | Accepted. The XSS fix was the most important one here. The app felt genuinely stable after this pass, though the code was still one big file doing too many things. |
| 4 | The core logic is working. Now refactor the backend into a proper module structure — config.py for env settings, models.py for schemas, services/run_service.py for business logic, and api/routes.py for the HTTP layer. | Provided the full split with clean file boundaries. main.py became a thin entry point that just wires everything together. | Accepted. This was the structure I already had in mind going in. It took a flat script and turned it into something a team could actually work on. |
| 5 | Do the same for the frontend. Move the API calls into client/runs.js, extract the polling into a custom useRunPoller.js hook, and split the UI into RunForm.jsx and RunResult.jsx. | Generated the modular component and hook structure, separating data fetching from presentation cleanly. | Accepted. The 200-line App.jsx was hard to reason about. After this split, each file had one clear job and the logic was easy to follow. |
| 6 | Look at the new structure and tell me what's missing before we can actually ship this. | Flagged three things: zero logging on the backend so we'd be flying blind in production, the frontend was still hardcoded to http://localhost:8000, and there was no /health endpoint for load balancers. | Accepted. All three were real gaps. Had it add structured logging with stack traces on failure, replace the hardcoded URL with an environment variable, and add a simple /health route. |
| 7 | Configure the frontend for Vercel and the backend for Render. | Produced a vercel.json with SPA rewrite rules, updated the Vite config for proper chunk splitting, and wrote a render.yaml blueprint with a Procfile for the backend. | Accepted with one fix needed. Render defaulted to a WSGI gunicorn command which immediately crashed the FastAPI app. Wrapped it in a Procfile explicitly using gunicorn with uvicorn workers, which is the correct setup for an ASGI app. Both services are now live and running. |
